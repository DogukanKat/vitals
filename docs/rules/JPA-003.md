# JPA-003 — `spring.jpa.open-in-view` is enabled or unset

| Field    | Value                                                                  |
| -------- | ---------------------------------------------------------------------- |
| Severity | Warn                                                                   |
| Category | JPA                                                                    |
| Since    | 0.1.0                                                                  |
| Module   | `vitals-rules-jpa`                                                     |

## What it checks

The rule scans every `application*.{yml,yaml,properties}` and `bootstrap*.*` file in the project. It fires once if either of the following is true:

- The property `spring.jpa.open-in-view` is set to `true` in any config file (and not overridden to `false` anywhere).
- The property is **not set** in any config file. Spring Boot's default is `true`, so silence is the same as opt-in.

If any config file sets the value to `false`, the rule stays quiet — the developer has clearly thought about it.

The diagnostic attaches to `application.yml` if present, otherwise `application.properties`, otherwise `bootstrap.*`.

## Why it matters

Open Session in View (OSIV) keeps a Hibernate `Session` and an underlying JDBC connection open for the entire HTTP request — through the controller, through the view rendering, until the response flushes. The convenience is real: lazy associations resolve transparently from JSP/Thymeleaf without `LazyInitializationException`.

The cost is real too. Every request — even those whose work is already done before the database is touched — holds a connection until the very last byte of the response. Under realistic load:

- The HikariCP pool (default size 10) saturates immediately. New requests block in `getConnection`.
- p99 latency goes from "fast" to "10 seconds, the HTTP timeout, then a 503".
- Slow downstream services (a logging sidecar, a slow Jackson serializer, an image CDN) hold DB connections while doing zero database work.

Hibernate and Vlad Mihalcea have argued against this default for years; Spring Boot has kept it for compatibility, but they print a warning on startup precisely because the team knows this is the wrong default for production. Disable it and resolve lazy associations explicitly — with a `JOIN FETCH`, an entity graph, or a DTO projection.

## How to fix

Set the property to `false` in your primary config:

```yaml
spring:
  jpa:
    open-in-view: false
```

…then fix the `LazyInitializationException` cases that surface. In a typical Spring Boot service:

1. Audit the endpoints that touched lazy associations in the view layer. They will now throw on the first read after the transaction commits.
2. Replace each lazy read in the controller / serializer with either an eagerly-loading query (`@EntityGraph` or `JOIN FETCH`) or a DTO projection that does its own joins.
3. Add an integration test that hits the endpoint with the relations actually exercised; the test will catch any remaining `LazyInitializationException`.

## When to suppress

Only when you genuinely need OSIV — a server-side templating layer (Thymeleaf, JSP) that depends on lazy loading and the team has accepted the connection-holding behavior. In that case, set the property to `true` explicitly so the intent is recorded in code review, and suppress this rule.

For REST APIs, there is no good reason to leave OSIV enabled.

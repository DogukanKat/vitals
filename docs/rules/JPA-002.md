## What it checks

The rule runs in two passes:

1. Scans every `@Entity` in the project and collects the getter names for fields annotated with `@ManyToOne`, `@OneToOne`, `@OneToMany`, or `@ManyToMany` (e.g. a field `customer` becomes `getCustomer`).
2. Scans every Java source for calls to those getters where the call is qualified on an instance (not `this`) and lives inside a loop body ŌĆö `for`, `for-each`, `while`, `do/while`, or a lambda passed to a stream per-element op (`forEach`, `map`, `filter`, `peek`, `flatMap`, `anyMatch`, ŌĆ”).

It's a heuristic. It will miss the case where the relation has already been `JOIN FETCH`-ed (we don't do query analysis), and it cannot tell whether the loop runs once or a million times. Both blind spots are acceptable: if you've already fetched the relation, the call is free; if the loop runs once, the rule fires but the cost is trivial.

## Why it matters

The N+1 query problem is the single most common JPA performance bug in production Spring Boot applications. The mechanism:

```java
List<Order> orders = orderRepository.findAll();        // 1 SELECT ŌĆ” FROM orders
for (Order order : orders) {
    String name = order.getCustomer().getDisplayName(); // N ├Ś SELECT ŌĆ” FROM customer WHERE id = ?
}
```

One query becomes `1 + N`. In integration tests against ~10 rows it looks fine. In production, `findAll` returns 5,000 orders, the customer table is on a different replica, and now every page render triggers 5,001 round-trips at ~2 ms each. Latency goes from 40 ms to 10 seconds; the HikariCP pool exhausts; the application returns 503s under load. This is the canonical "it worked on staging" outage.

The pattern is so common that Hibernate ships [`Hibernate.isPropertyInitialized`](https://docs.jboss.org/hibernate/orm/current/javadocs/org/hibernate/Hibernate.html#isPropertyInitialized(java.lang.Object,java.lang.String)) explicitly to help detect it, and Vlad Mihalcea's blog has an entire category dedicated to it. Catching it at static-analysis time ŌĆö before the query ever runs ŌĆö is the cheapest possible fix.

## How to fix

Pre-load the association as part of the initial query. Three idioms, in order of preference:

**`JOIN FETCH` in a query** (most explicit, easiest to review):

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @Query("select o from Order o join fetch o.customer")
    List<Order> findAllWithCustomer();
}
```

**Entity graph** (declarative; good when you have several flavours of "load with X"):

```java
public interface OrderRepository extends JpaRepository<Order, Long> {

    @EntityGraph(attributePaths = "customer")
    List<Order> findAll();
}
```

**Batch fetching** as a global default (`@BatchSize(size = 50)` on the association field or `hibernate.default_batch_fetch_size`): turns N+1 into `1 + ceil(N / 50)`. Use as a safety net, not as the primary fix.

## When to suppress

This rule has a higher false-positive rate than JPA-001. Suppress only when you can answer "yes" to both:

1. The relation is provably loaded before the loop ŌĆö `JOIN FETCH`, entity graph, or `@BatchSize` make this true.
2. The collection size is bounded by something other than user input ŌĆö a config flag, a small enum-like reference table.

Once suppression metadata lands (`@SuppressWarnings("vitals:JPA-002")` and a `// vitals:ignore` line comment), use it sparingly and keep a comment explaining which fetch strategy makes the suppression safe.

# TX-001 — Blocking I/O inside `@Transactional`

| Field    | Value                                                                  |
| -------- | ---------------------------------------------------------------------- |
| Severity | Error                                                                  |
| Category | Spring                                                                 |
| Since    | 0.1.0                                                                  |
| Module   | `vitals-rules-spring`                                                  |

## What it checks

The rule looks for three patterns invoked from within a `@Transactional` method (or any method on a class annotated `@Transactional`):

1. `Thread.sleep(...)` — direct match.
2. Method calls on receivers typed as `RestTemplate` or `RestClient`. The type is read from the declaration in the same source: fields, parameters, and local variables.
3. Reactor `.block()`, `.blockFirst()`, `.blockLast()`, `.blockOptional()` — matched by method name only.

The annotation is recognised whether it is written as `@Transactional`, `@org.springframework.transaction.annotation.Transactional`, or `@jakarta.transaction.Transactional`. The rule does no symbol resolution — a custom class with a `.block()` method that is not Reactor will produce a false positive. That is an acceptable trade-off: the fix is the same shape either way (move the call outside the transaction).

## Why it matters

A `@Transactional` annotation translates, at runtime, into "open a database connection, hold it from the first byte of the method to the last, then commit or roll back". The connection cannot serve any other request while the method is executing.

When the method then makes a network call:

- A synchronous `RestTemplate` request takes 100 ms on a good day, 5 s when the downstream service is degraded, and the full HTTP timeout (typically 30 s) when it has crashed.
- A `Thread.sleep(...)` does exactly what it says: holds the connection for the sleep interval, doing zero work.
- A Reactor `.block()` chain waits for an asynchronous pipeline to complete before letting the transactional method return.

With a HikariCP pool size of 10 (the Spring Boot default), it takes only 10 concurrent transactional requests blocked on a slow downstream call to lock out every other request in the application. The Tomcat thread pool then fills with threads waiting on `getConnection`, the load balancer marks the instance unhealthy, and you find yourself in an incident channel at 02:00.

This is one of the most common patterns behind "the database connection pool keeps exhausting under load" outage reports. Spring's own reference docs are explicit: a transaction should encompass the minimum work needed to maintain consistency, and certainly not external I/O.

## How to fix

Do the network call outside the transaction. Two idiomatic shapes:

**Fetch first, write second:**

```java
public Order placeOrder(NewOrder request) {
    TaxQuote quote = restTemplate.getForObject("/tax/" + request.id(), TaxQuote.class);
    return persistOrder(request, quote);
}

@Transactional
Order persistOrder(NewOrder request, TaxQuote quote) {
    Order order = orderRepository.save(request.toEntity());
    order.applyTax(quote);
    return orderRepository.save(order);
}
```

**Use Spring's `TransactionTemplate` for a narrow critical section** when you need to mix sync I/O and DB writes in one logical operation, and never inside the `execute` block.

For Reactor, prefer keeping the entire pipeline asynchronous. If a `.block()` is unavoidable (e.g. legacy boundary), run it on a dedicated bounded scheduler outside the transactional method.

## When to suppress

Effectively never. A real exception is calling a stubbed in-process service that completes in microseconds — but a fast in-process call doesn't need to be on a `RestTemplate` to begin with.

For tests, mark the test class as not transactional, or use a non-`@Transactional` orchestrating method.

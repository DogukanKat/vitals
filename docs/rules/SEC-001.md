# SEC-001 — Sensitive Actuator endpoint exposed over HTTP

| Field    | Value                  |
| -------- | ---------------------- |
| Severity | Error                  |
| Category | Security               |
| Since    | 0.1.0                  |
| Module   | `vitals-rules-spring`  |

## What it checks

The rule reads `management.endpoints.web.exposure.include` from every `application*.{yml,yaml,properties}` and `bootstrap*.*` in the project — accepting both the comma-separated scalar form (`include: "*"`, `include: health,info`) and the YAML list form (`include: [health, info, heapdump]`).

It fires on:

- The wildcard `*` — every Actuator endpoint, including the dangerous ones, becomes reachable over HTTP.
- Any explicit reference to a sensitive endpoint: `heapdump`, `shutdown`, `env`, `configprops`, `threaddump`, `httptrace`, `httpexchanges`.

It does **not** consult `management.endpoints.web.exposure.exclude`. If your `exclude` properly carves out the dangerous endpoints, suppress the rule and leave a comment in the config — but the safer choice is almost always to use a positive allowlist (`include: health,info,metrics`) and never the wildcard.

The rule cannot tell whether the endpoints sit behind a Spring Security filter chain. The diagnostic message asks the operator to verify the auth path.

## Why it matters

Each of these endpoints has been the entry point of a real, public incident:

- **`/actuator/heapdump`** — downloads the full JVM heap. Strings in memory at the time of the snapshot include database passwords, session cookies, JWT secrets, anything the application has touched in the last GC cycle. There is no way to redact a heap dump after the fact.
- **`/actuator/env`** — lists every environment variable and Spring property the application has resolved. In default configurations, values that look like secrets are masked — but only by simple suffix matching, and only on Spring Boot 2.x+. Custom property names commonly leak.
- **`/actuator/shutdown`** — POSTing to this endpoint stops the JVM. It is disabled by default; exposing it via `include: "*"` and forgetting to add `management.endpoint.shutdown.enabled=false` is a trivial DoS gift.
- **`/actuator/configprops`** — every `@ConfigurationProperties` bean's bound values. Same secret-leakage class as `/env`.
- **`/actuator/threaddump`** — stack traces of every thread, including method names of internal APIs. Useful for reconnaissance.
- **`/actuator/httpexchanges`** (formerly `/httptrace`) — the last N HTTP exchanges, headers included. Auth tokens, set-cookies, internal endpoints — all returned in JSON.

These have all appeared in public bug bounty reports against Spring Boot apps. The fix is one line of config.

## How to fix

Use an allowlist of the endpoints you actually need:

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus
```

`health` and `info` should be enough for most external probes. `metrics`/`prometheus` is fine for an internal scrape endpoint *if* it sits on a different port or network. The rest stay off.

If you genuinely need a sensitive endpoint:

1. Move Actuator to a separate management port (`management.server.port=9001`) bound to a non-routable interface.
2. Add a Spring Security filter chain that requires authentication for `/actuator/**`.
3. Restrict by IP at the load balancer / ingress layer.

Then suppress this rule with a config comment explaining the protection.

## When to suppress

When you have demonstrably-protected access — a separate management port behind a private VPC, or a security filter chain that requires authentication, and a code review approval that it is in place. The rule fires regardless because static analysis cannot prove these conditions; suppress with intent.

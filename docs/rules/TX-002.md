# TX-002 ‚Äî `@Transactional` on a private or protected method

| Field    | Value                  |
| -------- | ---------------------- |
| Severity | Error                  |
| Category | Spring                 |
| Since    | 0.1.0                  |
| Module   | `vitals-rules-spring`  |

## What it checks

The rule fires for any method declaration that:

1. Carries an `@Transactional` annotation (Spring's `org.springframework.transaction.annotation.Transactional` or Jakarta EE's `jakarta.transaction.Transactional`, qualified or not), **and**
2. Has visibility `private` or `protected`.

Package-private (default-access) methods are not flagged by this rule today. They sit in a CGLIB-specific gray area and warrant their own discussion; if you want them flagged, expect a follow-up rule.

The check is purely structural ‚Äî no symbol resolution.

## Why it matters

Spring implements `@Transactional` as proxy-based AOP. At startup, the container wraps each bean in a proxy (JDK dynamic proxy when the bean implements an interface, otherwise a CGLIB subclass) whose generated methods open a transaction, delegate to the real method, and commit or roll back on exit.

That proxy can only intercept calls that *go through it*. Two kinds of call bypass it:

- A call from outside the bean to a non-public method ‚Äî the proxy doesn't expose those methods, so the call resolves to the target object directly.
- A call from inside the bean to one of its own methods via `this.foo()` ‚Äî the call never crosses the proxy boundary.

When the call bypasses the proxy, the `@Transactional` annotation is silently ignored. The annotated method still runs; it just runs *outside any transaction*. There is no warning, no exception, and no test will fail unless someone has explicitly written one that asserts transactional behavior (which is rare). In production this typically surfaces as:

- Partial writes after an exception that *would* have rolled back if there had been a transaction.
- Race conditions on values that *would* have been protected by `@Transactional(isolation = ...)`.
- `LazyInitializationException` thrown in code that "looked transactional" but wasn't.

The Spring reference documentation is explicit: "the `@Transactional` annotation is typically used on methods with public visibility". Anything else is a bug ‚Äî usually an honest one, introduced when a developer extracted a helper and left the annotation on it.

## How to fix

The simplest fix is to make the method public:

```java
// Before
@Transactional
private Order persist(NewOrder request) { ... }

// After
@Transactional
public Order persist(NewOrder request) { ... }
```

If the method must remain non-public for visibility reasons, extract it onto a separate Spring bean and inject that bean. The proxy boundary will then be crossed naturally:

```java
@Service
public class OrderPersistence {
    @Transactional
    public Order persist(NewOrder request) { ... }
}

@Service
public class OrderService {
    private final OrderPersistence persistence;
    public Order placeOrder(NewOrder request) {
        return persistence.persist(request);
    }
}
```

A more invasive alternative is to enable AspectJ-mode transactional weaving (`@EnableTransactionManagement(mode = AdviceMode.ASPECTJ)`). That removes the proxy restriction entirely but adds a load-time-weaving step and is rarely worth it for this case.

## When to suppress

Effectively never. If you have what looks like a legitimate use case, prefer the "extract onto its own bean" fix ‚Äî it costs nothing at runtime and makes the transactional boundary visible in code review.

The one defensible exception is generated code that you do not own (e.g. MapStruct-generated mappers) ‚Äî but those should not be carrying `@Transactional` to begin with.

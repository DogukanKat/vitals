# JVM-001 — Java container image without an explicit heap bound

| Field    | Value               |
| -------- | ------------------- |
| Severity | Warn                |
| Category | JVM                 |
| Since    | 0.1.0               |
| Module   | `vitals-rules-jvm`  |

## What it checks

The rule scans every Dockerfile in the project (`Dockerfile`, `Dockerfile.*`, `*.Dockerfile`). It fires once per file that is both:

1. **A Java image** — a JDK/JRE base image (`eclipse-temurin`, `amazoncorretto`, `openjdk`, `ibm-semeru`, `sapmachine`, `azul/zulu`, `liberica`, `graalvm`, …) or a `java` invocation in `ENTRYPOINT`/`CMD`/`RUN`, **and**
2. **Missing a heap bound** — no `-Xmx` and no `-XX:MaxRAMPercentage` on any line (including `ENV JAVA_TOOL_OPTIONS` / `JAVA_OPTS` and the entrypoint command).

### A scoping note (deviation from the literal spec)

The Tier-1 spec phrases this as "container memory limit set but JVM heap not configured". Container memory limits are declared at orchestration time — Kubernetes `resources.limits.memory`, `docker run --memory`, Compose `mem_limit` — not in the Dockerfile, and frequently in a repository the analyzer never sees. Requiring the limit and the heap config to co-locate would make the rule almost never fire.

So JVM-001 keys off the Dockerfile: *if you ship a JVM in a container, you will run under a memory limit, so the heap must be sized intentionally.* A manifest-aware companion rule (reading k8s/Compose limits and correlating with `JAVA_TOOL_OPTIONS`) is reasonable future work; it is deliberately out of scope here.

## Why it matters

Before JDK 10, the JVM read host memory, not the cgroup limit. A 512 MiB pod on a 64 GiB node sized its heap for 64 GiB and was `OOMKilled` within seconds of load. That era is over — Java 21 has `UseContainerSupport` on by default and honours the cgroup limit.

The bug did not disappear, it changed shape. With no explicit sizing, Java 21 applies `MaxRAMPercentage=25`. In a 512 MiB container that is a ~128 MiB heap:

- A typical Spring Boot service does not finish starting in 128 MiB — it dies in a `GC overhead limit exceeded` / `OutOfMemoryError` loop during context refresh, which reads in production as a crash-looping pod that "worked on my machine" (where it had gigabytes).
- A 4 GiB pod with the same default wastes 3 GiB: the heap caps at 1 GiB and the rest of the limit is unused, so you pay for memory the JVM refuses to touch.

Either way the sizing is accidental. The fix is to make it a decision: pick a percentage (or an absolute `-Xmx`) appropriate to the workload and the limit.

## How to fix

Set a container-relative heap target — percentage scales with whatever limit the orchestrator assigns:

```dockerfile
ENV JAVA_TOOL_OPTIONS="-XX:MaxRAMPercentage=75.0"
```

`75.0` is a common starting point for a service container whose only significant process is the JVM (leave headroom for thread stacks, metaspace, and native buffers — do not use 100). Tune against real heap-usage metrics.

If the container runs a fixed, known limit, an absolute bound is equally valid and more predictable:

```dockerfile
ENTRYPOINT ["java", "-Xmx384m", "-jar", "/app.jar"]
```

Then verify under load that the chosen value leaves enough non-heap memory that the kernel does not OOM-kill the container.

## When to suppress

- **Multi-process or non-service images** where the JVM is incidental (a CLI tool invoked ad hoc, a one-shot migration job with a generous limit). The percentage default is harmless there.
- **Base images your platform team controls** that inject `JAVA_TOOL_OPTIONS` at runtime via an entrypoint script the analyzer cannot see. Confirm it actually does, then suppress with a comment.

For a long-running service container, there is no good reason to leave heap sizing to the default.

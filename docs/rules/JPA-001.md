# JPA-001 — `FetchType.EAGER` on `@ManyToOne` / `@OneToOne` / `@ManyToMany`

| Field    | Value                                                                  |
| -------- | ---------------------------------------------------------------------- |
| Severity | Error                                                                  |
| Category | JPA                                                                    |
| Since    | 0.1.0                                                                  |
| Module   | `vitals-rules-jpa`                                                     |

## What it checks

This rule fires when a field is annotated with `@ManyToOne`, `@OneToOne`, or `@ManyToMany` **and** the annotation explicitly sets `fetch = FetchType.EAGER`. Implicit-EAGER defaults (no `fetch` attribute on `@ManyToOne`/`@OneToOne`) are intentionally out of scope here; a separate rule will address those once we can also detect a corresponding `JOIN FETCH`.

## Why it matters

`FetchType.EAGER` says "every time you load this entity, load its association too — in a separate SQL statement if necessary". That sounds harmless in tests. In production it manifests as:

- **N+1 query storms.** A list endpoint that loads 200 `Order` rows triggers 200 extra `SELECT * FROM customer WHERE id = ?` round-trips, each waiting on network latency to the database. Tail latency goes from 40 ms to 1.5 s and the DB connection pool starves.
- **Transitive blow-up.** EAGER on `Order.customer` plus EAGER on `Customer.address` plus EAGER on `Address.country` means loading one order pulls four tables you didn't ask for. Hibernate cannot inline more than `hibernate.max_fetch_depth` levels of joins, so the rest become extra queries.
- **`MultipleBagFetchException`.** Two EAGER `@OneToMany` collections on the same entity will refuse to load at runtime in Hibernate ≥ 5.

This is the single most common cause of "the staging cluster fell over under realistic data" incidents in Spring Boot / JPA codebases. Vlad Mihalcea has written extensively about the pattern; the short version is: **default to `LAZY`, fetch what you need with a query.**

## How to fix

Switch the association to `LAZY` and load the relation explicitly when you need it.

```java
// Before
@ManyToOne(fetch = FetchType.EAGER)
private Customer customer;

// After
@ManyToOne(fetch = FetchType.LAZY)
private Customer customer;
```

Then, where you need the customer alongside the order:

```java
@Query("select o from Order o join fetch o.customer where o.id = :id")
Optional<Order> findWithCustomer(@Param("id") Long id);
```

Or use an entity graph for the call sites that need it:

```java
@EntityGraph(attributePaths = "customer")
Optional<Order> findById(Long id);
```

## When to suppress

Almost never. Two reasonable exceptions:

1. **The association is genuinely 1:1 and always required**, *and* the parent is never loaded in collections. Even then, prefer `LAZY` plus `JOIN FETCH` — the rule annotation `@SuppressWarnings("vitals:JPA-001")` will work once suppression support lands.
2. **Reference data backed by a second-level cache.** If the association is cached, the EAGER cost is local memory rather than a SQL round-trip. Document the cache configuration alongside the suppression.

If you find yourself reaching for a suppression more than once in a project, treat it as a signal to revisit the data model rather than to silence the rule.

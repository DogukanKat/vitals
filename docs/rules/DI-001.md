# DI-001 — `@Autowired` on a field (prefer constructor injection)

| Field    | Value                  |
| -------- | ---------------------- |
| Severity | Warn                   |
| Category | Spring                 |
| Since    | 0.1.0                  |
| Module   | `vitals-rules-spring`  |

## What it checks

The rule fires for any field declaration annotated with Spring's `@Autowired` (unqualified or fully-qualified as `org.springframework.beans.factory.annotation.Autowired`).

It does **not** fire for `@Autowired` on constructors (good — and in fact unnecessary on a single constructor in Spring 4.3+) or on setter methods (setter injection — a separate pattern). Only field-targeted annotations trigger the rule.

## Why it matters

Field injection produces an object that is unfinished at the end of its constructor. Every dependency is `null` until the framework reaches in and assigns it via reflection. This causes four concrete problems:

- **The class is no longer instantiable without Spring.** A unit test that previously read `new UserService(repo, notifier)` now needs a Spring context (or hand-injected reflection) to populate the dependencies. The test footprint balloons.
- **Mandatory dependencies look optional.** A reader of the code cannot distinguish "always present" from "wired only in some profiles" without launching the application. Constructor parameters carry the contract on their face.
- **Immutability is impossible.** Field injection requires the field to be non-`final`. Once everything is wired the field never legitimately changes, but the type system can't say so. Bugs that mutate the field at runtime (rare but real — usually in test setup) compile.
- **Circular dependencies are easier to introduce.** Spring will happily wire `A.b = B` and `B.a = A` even when no construction order resolves them. With constructor injection the cycle fails at startup with a clear stack trace.

The Spring team has documented this preference for years; the recommendation is explicit in the reference manual and reinforced by every modern Spring Boot starter, which uses constructor injection without exception.

## How to fix

Replace the annotated field with a `final` field and a constructor that takes the dependency:

```java
// Before
@Service
public class UserService {
    @Autowired
    private UserRepository userRepository;

    @Autowired
    private NotificationService notificationService;
    ...
}

// After
@Service
public class UserService {
    private final UserRepository userRepository;
    private final NotificationService notificationService;

    public UserService(UserRepository userRepository, NotificationService notificationService) {
        this.userRepository = userRepository;
        this.notificationService = notificationService;
    }
    ...
}
```

In Spring 4.3+, a single-constructor class needs no `@Autowired` on the constructor — Spring will use it automatically.

If you need an optional dependency, declare it as `Optional<T>` in the constructor parameter list, not as a nullable field.

## When to suppress

Effectively never in production code. Two practical exceptions:

1. **Tests.** Test classes are not under this rule's pressure to be unit-testable — they are unit tests. Field injection in `@SpringBootTest` is common and acceptable. (A future rule will scope this exception explicitly; for now, suppress with a comment.)
2. **Legacy abstract base classes** in a framework you cannot modify. Even there, prefer template-method patterns over inherited injected state.

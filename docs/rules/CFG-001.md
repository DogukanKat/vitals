## What it checks

For every `application*.{yml,yaml,properties}` and `bootstrap*.*` in the project, the rule inspects each property whose final dot-segment names a credential:

`password`, `passwd`, `secret`, `apikey`, `api-key`, `token`, `access-token`, `refresh-token`, `private-key`, `secret-key`, `access-key`, `client-secret`, `credential`, `credentials` â€” plus any segment ending in `password` or `secret` (so `key-store-password`, `trust-store-password`, `client-secret` are covered).

A finding is raised when the value is:

- non-blank, **and**
- not a Spring property placeholder (does not start with the dollar-brace form), **and**
- not an obvious non-secret literal (`true`, `false`, `none`, `null`, `disabled`, `enabled`, `default`).

Matching on the *final segment* keeps the false-positive rate low: `app.feature.password-rotation-enabled=false` is not flagged because its segment is `password-rotation-enabled`, not `password`, and the value is a boolean anyway.

## Why it matters

A secret written into `application.yml` or `application.properties` is a secret committed to version control. Once it lands in git history it is effectively public to everyone with repository access â€” and unrecoverable: rotating the credential is the only remediation, because `git filter-repo` cannot reach forks, clones, CI caches, or the screenshots in a Slack thread.

The concrete failure modes:

- **Repository leak.** A public push, an over-broad fork, a contractor offboarded without access revocation â€” any of these turns a checked-in production password into an incident.
- **Environment bleed.** Hardcoded values do not vary per environment. The classic outcome is staging credentials reaching production, or the reverse: prod credentials running in a developer's laptop test.
- **No rotation story.** Secret rotation becomes a code change, a PR, a deploy â€” so in practice it never happens, and the credential lives for years.

Externalized configuration solves all three: the value lives in an environment variable, a Kubernetes secret, or a vault, and the config file carries only a placeholder that resolves at startup.

## How to fix

Replace the literal with a placeholder and supply the value from the environment:

```yaml
# Before
spring:
  datasource:
    password: S3cr3t-Prod-Pw

# After
spring:
  datasource:
    password: ${DB_PASSWORD}
```

```properties
# Before
app.payment.api-key=sk_live_51HxYz...

# After
app.payment.api-key=${PAYMENT_API_KEY}
```

Then provide `DB_PASSWORD` / `PAYMENT_API_KEY` via the platform's secret mechanism (Kubernetes `Secret`, Docker secret, AWS Parameter Store, HashiCorp Vault, or `.env` in local dev â€” and `.env` is git-ignored).

If a secret has already been committed, rotate it. Removing it from history is necessary but not sufficient â€” assume it is compromised.

## When to suppress

- **Non-secret values that match the key heuristic.** Example: `spring.kafka.properties.security.protocol` is not a secret but contains no secret either; this rule already ignores it because the segment is `protocol`. If you hit a genuine false positive, suppress with a comment.
- **Test fixtures and throwaway local defaults** that are demonstrably not real credentials (`password: test`). Even then, prefer `${DB_PASSWORD:test}` so the shape is consistent.

There is no defensible reason to suppress this rule for a real production credential.
